# Design: Classroom Recovery Card

Generated by Codex Office Hours on 2026-04-26 22:59:18 CST
Status: DRAFT
Mode: Startup

## Problem Statement

FocusMate should not start as a generic AI note-taking, meeting summary, or classroom transcript tool.

The first product wedge is narrower:

> A university student loses classroom context because they drifted off, switched screens, checked something else, or could not process the previous minute fast enough. When the teacher asks a question, assigns a task, or changes discussion direction, the student needs to know what they just missed quickly enough to rejoin the class.

The user-facing job is:

> "What did I just miss?"

## Demand Evidence

The initial seed demand comes from the user's own classroom pattern:

- The current high-pain setting is class, especially courses such as Comprehensive English.
- A class block is roughly 45 minutes plus 45 minutes.
- The need appears when the teacher stops and asks a question or assigns a task.
- The current behavior after missing context is often to give up and keep listening without recovering the missed context.

This is real pain, but not yet market proof. The next validation step is repeated personal use in live classes, not broad positioning.

## Status Quo

The current fallback is:

- Miss the prior context.
- Do not interrupt the teacher or ask classmates.
- Keep listening and hope the later context becomes clear.

This matters because the replacement product must be faster and easier than doing nothing. If the recovery card takes too long, the user will revert to the old behavior.

## Target User and Narrowest Wedge

Target user:

> University students who want to stay engaged in class but periodically lose context, especially when the teacher asks a question, assigns a task, or switches discussion direction.

Seed course types:

- Comprehensive English and other language-heavy classes.
- Bilingual or foreign-language courses.
- Large lecture or general education classes where the student may drift, but still wants to recover when interaction is required.
- Other courses with explicit classroom questions or task moments.

The first wedge is not "all students" and not "all note-taking." It is:

> Manual, in-class recovery at the moment the student realizes they missed something.

## Constraints

- The first version must be pure manual trigger.
- No automatic interruption or zero-click prompt in MVP.
- The product must produce something useful in 3-8 seconds.
- Full transcript can arrive later or remain collapsed.
- The main screen must not become a live transcript stream.
- The system should use recent transcript buffers rather than reprocessing the whole class.

## Premises

1. The target user is a university student who wants to return to classroom context when they realize they missed something.
2. The core pain is in-class recovery, not post-class review.
3. The recovery window is short. FocusMate must return a good-enough card in 3-8 seconds, then improve or fill evidence later.
4. The main value is a recovery card. Transcript is an evidence layer, not the product surface.
5. Manual trigger is the right first test because it validates whether the recovery card itself is valuable before adding attention detection.

## Approaches Considered

### Approach A: Manual Recovery Card MVP

User taps `What did I just miss?` and FocusMate summarizes the last 60 seconds into a Chinese recovery card. The original transcript is collapsed by default.

Effort: S
Risk: Low

Pros:

- Fastest path to validating the core value.
- Avoids false positives from automatic attention detection.
- Fits the live classroom setting with low interruption risk.

Cons:

- Requires the user to remember to tap.
- Does not yet feel like a full attention agent.
- Does not validate automatic teacher-question detection.

Reuses:

- Existing browser microphone capture.
- Existing WebSocket audio stream.
- Existing DashScope transcription path.

### Approach B: Classroom Buffer With Multiple Time Windows

FocusMate maintains rolling transcript buffers for 30 seconds, 60 seconds, and 3 minutes. The user chooses the recovery window before generating the card.

Effort: M
Risk: Medium

Pros:

- Better fits real missed-context durations.
- Allows measuring which window users actually need.
- Creates a foundation for later smart prompts.

Cons:

- Adds UI and state complexity.
- May distract from validating the core card.
- Longer windows increase latency and model cost.

Reuses:

- Existing live transcript stream.
- A rolling text buffer on the client or server.

### Approach C: Question and Task Signal Extraction

The system extracts whether the recent transcript includes a teacher question, assignment, instruction, or key point. These signals are highlighted in the recovery card.

Effort: M
Risk: Medium

Pros:

- Directly matches the user's stated trigger moment.
- Makes the card more actionable than a normal summary.
- Can later evolve into a light automatic prompt.

Cons:

- Adds model or rule-based classification work.
- Mistakes could reduce trust.
- Background analysis may increase latency or cost if implemented too early.

Reuses:

- Existing transcript text.
- Same prompt path as card generation.

## Recommended Approach

Start with Approach A and include the smallest useful part of Approach C.

The first version should be a manual recovery card, with the card template already designed around question/task/key-point extraction. Do not build background detection yet.

Default window:

- 60 seconds.

Later window options:

- 30 seconds.
- 3 minutes.

First card shape:

```text
我刚刚错过了什么？

刚刚讲到：
...

现在要做什么：
如果老师刚刚提问、布置任务、给出要求，在这里直接写。
如果没有检测到明确任务，写“未检测到明确任务”。

你现在该接着听：
...

原文逐字稿
[展开]
```

## Open Questions

- In real class use, how often does the user tap the recovery button per 45-minute block?
- Is 60 seconds the right default window, or does 30 seconds recover most cases?
- Does the recovery card need to be in Chinese by default for all courses, or only language-heavy courses?
- Should the "now do this" section prioritize questions, assignments, or conceptual transitions?
- What is the acceptable cost and latency per recovery card?

## Success Criteria

The MVP is worth continuing if, across real classroom use:

- The user taps the recovery button at least 3 times across a 45+45 minute class block.
- At least half of generated cards are judged useful enough to continue listening.
- First useful content appears within 3-8 seconds.
- The user prefers the recovery card over simply giving up and hard-listening afterward.
- The transcript is used for verification, not as the primary interaction surface.

## Distribution Plan

Initial distribution should be private and local:

- Use the existing web demo on the user's laptop or phone.
- Run in real classes with microphone permission.
- Save no public recordings by default.

After personal validation:

- Share with 3-5 classmates who take similar classes.
- Ask them to use it in one live class and report every time they tapped the recovery button.
- Compare actual tap moments with perceived usefulness.

## Dependencies

- Reliable low-latency transcription from live classroom audio.
- Rolling transcript buffer for recent 60 seconds.
- A fast model prompt for Chinese recovery card generation.
- UI that keeps the recovery button available without becoming distracting.
- Clear privacy behavior around classroom recording and transcript storage.

## The Assignment

Build the smallest live-class prototype that can answer one question:

> When the user taps "我刚刚错过了什么？", can FocusMate return a useful recovery card within 3-8 seconds?

Do not implement automatic prompts, full class notes, user policies, or long-term memory until this is proven.

## What I Noticed

- The strongest phrasing was "我刚刚错过了什么". That is much sharper than "Attention Infrastructure" for an MVP.
- The key trigger is not generic distraction. It is the moment a teacher asks, assigns, or expects the student to rejoin.
- The current status quo is giving up and hard-listening. That sets a high bar for speed because doing nothing is instant.
- The product should earn the larger agent vision by first proving one recovery moment.
